# GATE_CONTROL_MODEL.md

## Core Principle

Gates define reality progression conditions.
Dashboard surfaces and enforces those conditions.

**Non-negotiable rule:**
> If the gate describes software instead of reality, it is wrong.

---

## Operating Relationship

- **Gates = physics (truth conditions for progression)**
- **Dashboard = instrumentation (visibility + enforcement)**

Each gate definition must include only:
1. **Required reality conditions** (what must be true)
2. **Required visibility** (what must be observable)
3. **Required enforcement checks** (what must block if false)

---

## USG Initiative Sequence — Gate-Aligned Control Model

### Gate 0 — Platform Entry
**Required reality conditions**
- Platform pressure confirmed
- Capital presence confirmed
- FID definability confirmed
- Multi-actor dependency confirmed

**Required visibility**
- Signal validation outcome (real vs noise)
- Platform GO / NO-GO

**Required enforcement checks**
- Block initiative entry if any required condition is unverified

---

### Gate 1 — Vision Intake
**Required reality conditions**
- Platform defined
- Buyer defined
- Structural blockage defined

**Required visibility**
- Gate stage = 1
- Buyer ↔ initiative linkage
- One-line structural truth
- Stakeholder map

**Required enforcement checks**
- Block Gate 1 completion if any required definition is missing

---

### Pre-Mandate Phase (not a gate)
**Required reality conditions**
- Buyer engagement state visible
- Structural pressure level visible
- Mandate readiness signal visible

**Required visibility**
- Meeting context (non-progress artifact)
- Buyer status (exploring / recognizing / aligned)
- Primary constraint explicitly set to no-mandate condition

**Required enforcement checks**
- Block any Gate 2+ advancement while mandate condition is false

---

### Mandate Phase (hard entry condition)
**Required reality conditions**
- Mandate signed
- Fee secured

**Required visibility**
- `mandate_signed` = true/false
- `fee_secured` = true/false

**Required enforcement checks**
- No mandate → no Gate 2+

---

### Gate 2 — Information Intake
**Required reality conditions**
- Information completeness threshold met
- Known unknowns enumerated
- Access level defined

**Required visibility**
- Transition into Gate 2 recorded
- Data completeness metric
- Missing information list

**Required enforcement checks**
- Block Gate 3 if unknowns are unbounded/critical

---

### Gate 3 — Governance Review
**Required reality conditions**
- Capital stack defined
- Decision rights defined
- Risk allocation defined
- Promote structure defined (not negotiated pre-mandate)

**Required visibility**
- Governance summary
- Actor alignment score
- Promote status
- Decision authority map

**Required enforcement checks**
- Block Gate 4 if governance package is incomplete

---

### Gate 4 — Workstream Conversion
**Required reality conditions**
- Workstreams defined
- Scope boundaries explicit
- Dependencies mapped

**Required visibility**
- Workstream breakdown
- Dependency map
- Explicit exclusions

**Required enforcement checks**
- Block Gate 5 if scope/dependencies remain ambiguous

---

### Gate 5 — Operator Assignment
**Required reality conditions**
- Operator assigned
- Execution ownership externalized (not USG)

**Required visibility**
- Operator identity
- Pod/ownership structure
- Delivery entities

**Required enforcement checks**
- Block Gate 6 if execution ownership is unresolved or internalized

---

### Gate 6 — Sprint Execution
**Required reality conditions**
- Deliverables advancing
- Active blockers identified
- Timeline integrity maintained

**Required visibility**
- Sprint status
- Task progress
- Active constraints

**Required enforcement checks**
- Block Gate 7 if critical artifacts are incomplete

---

### Gate 7 — Final Review
**Required reality conditions**
- System completeness achieved
- Quality bar passed

**Required visibility**
- Final readiness
- Approval decision (approve/reject)

**Required enforcement checks**
- Block post-gate activation if final review is not approved

---

### Post-Gates — Activation
**Required reality conditions**
- FID completed
- Promote activated
- Execution externalized

**Required visibility**
- FID state
- Promote state
- Execution ownership state

**Required enforcement checks**
- Raise exception if execution ownership regresses to internal

---

## Model Integrity Rules

1. Gates describe reality, not product features.
2. No state progression without evidence-backed transition.
3. No Gate 2+ without mandate signed and fee secured.
4. One primary constraint + one next required transition per initiative.
5. If a condition is not visible, it is treated as not true.
