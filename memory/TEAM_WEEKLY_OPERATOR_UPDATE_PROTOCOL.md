# TEAM_WEEKLY_OPERATOR_UPDATE_PROTOCOL.md

## Purpose
Use this protocol for **all emails to Tyreek, Richard, Maryan (CC Isaac)** so team communication enforces decision cadence, not passive status reporting.

## Non-Negotiable Spine (no variation)

### 1) Subject (always)
`Weekly Initiatives — State, Constraints, Next Moves`

### 2) Opening (minimal, consistent)
Hi [Team] —

Weekly snapshot of active initiatives.
Focus: where each stands, what’s blocking it, and what moves next.

### 3) Active Initiatives Block (core)
Only include active, real initiatives.

Format for each initiative:

`[Initiative Name]`
- `State: [current state]`
- `Constraint: [single biggest blocker, in plain operator language]`
- `Next Move: [single actionable move, in plain operator language]`

Rules:
- Exactly **one** constraint per initiative
- Exactly **one** next move per initiative
- No paragraphs or explanation text in this section
- **Do not use system IDs/codes** in the email body for `Next Move` (e.g., avoid `st_req_*`)
- Every line must be instantly understandable without translation

### 4) Required Actions block (always)
`This Week — Required`
1. Confirm owner per initiative
2. Confirm the single constraint is correct
3. Confirm next move + what proves completion

### 5) Optional Changes block (only if needed)
Use only when true:
- New initiative added
- Initiative removed
- State change milestone reached

If no changes, omit this block.

### 6) Close (always short)
`I’ll send transition cards once owners are confirmed.`

`— Sentinel`

## Message-Length Constraints
- Keep to ~1 mobile screen
- Include ~5–7 initiatives max

## Constraint + Next Move Quality Standard (anti-gap rules)
Before sending, validate each initiative line against these checks:
1. **No translation required**: a human operator can understand constraint + next move in <5 seconds.
2. **Constraint is concrete**: must describe a specific wall (missing owner, missing commitment, missing approval, missing capital definition, etc.), not abstract phrasing.
3. **Next move is executable this week**: should be assignable, trackable, and finishable.
4. **No system-language leakage**: do not expose internal IDs (`st_req_*`, object keys) in email body.
5. **One-screen clarity**: if a line needs explanation, rewrite until it does not.

## Behavioral Intent
This cadence must train:
1. Constraint thinking (single blocker)
2. Transition thinking (single next move)
3. Ownership clarity (explicit owner confirmation)

## Consistency Rule
Never skip a weekly send. Consistency is part of the operating system.
